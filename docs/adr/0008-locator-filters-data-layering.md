# 位次定位结果过滤的数据来源与分层

位次定位结果页加多维过滤：**省份、院校层次（985/211/双一流）、办学性质（公办/民办）、学校类别（综合/理工/师范…）、城市层级、专业大类（学科门类）、专业关键词、计划人数、组内专业数、隐藏中外合作及高收费**。本 ADR 记录这些过滤维度的数据从哪来、放哪。

**两层存储，避免撑大客户端索引**。`locator-wuli.json` 已 2.6 MB / 1.1 万行、院校名与代码重复约 11 次。故把过滤数据按粒度分两处：

- **院校级属性 → 新增 `public/data/school-meta.json`**，按院校代码建键（约 1077 条，紧凑键），客户端**一次性 fetch、按 `sc` 挂接**。承载：省份、城市、城市层级、办学性质、学校类别、层次（985/211/双一流）。
- **逐专业属性 → 加进 locator 索引每行**（无法去重到院校）：`门类`（1 字码）、`学费`、`中外合作`（布尔，false 省略）。计划人数、组内专业数、选科已在索引内。

**来源**：省份/城市/办学性质/学校类别 取自 2020–2023 万师兄旧格式表（与 985/211 同源，扩展 `internal/hlj/tags.go`，按校名挂接，覆盖约 91%）；学费/中外合作取自 2026 招生计划（中外合作按专业名/全称/备注含「中外·合作办学」判定，约 664 个）；阈值「隐藏」= 中外合作 或 学费 ≥ 20000（`待定` 不隐藏）。

**两个"自建"数据，因官方/三方源无对应列**（记下来，免得后人当 bug "修"）：

- **专业大类 = 12 学科门类，靠自建 `专业名→门类` 映射**。新科类（物理/历史）的专业分数线与 2026 招生计划**都没有门类列**，仅旧科类 2020/2021 表与 200 行专家版样本有。故在 Go 内维护一张归一化专业名→门类的映射表，未命中归「其他」并 `log` 出数量。选 12 门类（而非 ~90 专业类或粗分桶）是因为考生/家长就按门类思考，且 12 项适合多选。
- **城市层级（一线/新一线/二线…）= 静态 `城市→层级` 映射**（约 50 城）。来源表无城市层级列（只在样本里有）。

**过滤语义**：维度之间 AND、单个多选维度内部 OR；省份多选、关键词单值。过滤项与"我的位次/选科"一并存 localStorage，并在结果上方以可移除的 chip 显示当前生效过滤 + 一键清除，避免"为什么结果被筛过"的困惑。

**取舍**：选边表 join 而非把院校属性铺进每行——索引保持精简、院校属性变成一张可缓存小表；代价是客户端要做一次按 `sc` 的挂接。门类与城市层级自建，是已知的维护成本，换来全覆盖的可用过滤。
