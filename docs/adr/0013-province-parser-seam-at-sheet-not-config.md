# 省份解析的缝放在「表头感知的 sheet 读取器」，而非一个共用大配置

黑龙江与浙江的 xlsx 解析器（专业录取分数线、招生计划）看起来高度重复，约 190 LOC、
约 70% 同形。直觉是抽一个 `core.ParseAndAggregate(rows, config)`，让两省只提供「列名 /
科类过滤 / 批次过滤 / 名称归一」的配置。**本 ADR 否决这个方向，并记录真正下沉的是什么。**

## 真正共用的 vs 真正因省而异的

聚合早已共用：`core.AggregateLeaves` 把行表聚合成院校×专业叶子，两省都调它。管线是
`省解析器 → []core.MajorScoreRow → core.AggregateLeaves`——聚合不重复。

剩下「重复」的其实只有两段：

1. **真正同形、纯样板**：开文件 → 取首个 sheet → 读所有行 → 按谓词定位表头 → 建列定位闭包。
   每个解析器开头都照抄一遍。
2. **看着像、其实因省而异**：逐行的过滤 + 字段抽取 + 装配。黑龙江按 `物理/历史` 集合 +
   含「本科」过滤、装 `PlanRow`（有专业组代码/全称）；浙江按 `==综合` + `batchKeep`、院校代码
   补零、大类按方向拆分、装 `PlanRow2026`（有招生类型、无组）。过滤谓词、字段变换、**输出结构**
   都不同。

## 决定

只下沉第 1 段：新增 `core.Sheet` + `core.OpenSheet(path, isHeader)` / `core.NewSheet(rows, isHeader)`，
把「文件 I/O + 首 sheet + 表头定位 + 列定位」收进一个深模块。小接口（给一个「哪行是表头」的谓词，
拿回数据行游标 + `Col()`），大实现（文件生命周期、sheet 选择、空表检查、表头扫描、模糊列匹配）。
被 4 个解析器（+ 未来更多）复用。

第 2 段**留在各省解析器**。把它塞进共用配置会得到一个**浅模块**：配置/回调会和它取代的重复一样复杂
（每省要传过滤谓词 + 字段映射 + 一个装配回调，等于把循环体外移），删除测试不成立——复杂度只是从
循环挪进配置，没有被消除。这恰恰是本仓「深模块」原则要避免的。

## 给未来的人

不要「好心」把两省的逐行解析循环合并成一个 `ParseAndAggregate(config)`。那看起来更 DRY，实则用
一个浅配置换掉了诚实的省份专属代码。缝的正确位置是 `core.Sheet`（真正同形的 I/O 样板），不是
解析循环（真正因省而异）。`internal/<省>` 只装省份专属逻辑，是有意为之。
