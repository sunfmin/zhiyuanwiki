# 信息架构与位次定位交互

**首页三入口并列**：位次定位 / 院校浏览 / 专业浏览，三条对等的门。因此**专业浏览进入 MVP**——意味着专业需要一套跨校、跨年的归一化规则（专业名清洗、"专业类 vs 具体专业"、大类招生），这是专业轴的主要开放设计问题。

**URL：稳定校代码路径**。院校页 `/yuanxiao/{校代码}/`，用稳定校代码（跨改名稳定、不怕重名、URL 短）。放弃了拼音 slug 与中文直入 URL。

**院校×专业叶子 = 院校页内的锚点区块，不再是独立文件**。原计划每个 (院校,专业) 一个独立页 `/yuanxiao/{校代码}/zhuanye/{专业键}/`，但实测部署发现 Cloudflare Pages 单次部署有 **2 万文件硬上限**，而院校×专业叶子有近 2 万个，加上院校/专业页会超限。故把每个专业的历年走势折叠进院校页，做成可展开的锚点区块 `/yuanxiao/{校代码}/#z-{专业键}`（专业键 = 归一化专业名的确定性短哈希，因专业代码与组号一样逐年变，不能做稳定标识）。这样全站文件数降到约 2 千，留足余量，且仍是纯静态（不引入 SSR）。定位结果与专业页均链接到该锚点。

**数据可用年份**：黑龙江 2024 首届新高考；专业录取分数线中 2023/2024/2025 为新科类口径（物理/历史 + 院校代码 + 最低位次），可直接进位次模型；2020–2022 为旧科类（理科/文科）或缺位次，**不混入**物理/历史走势。

**位次定位 island**：输入**分数或位次**（输入分数时按当年一分一段自动换算成位次）+ 科类 + 选科 → 输出**院校×专业逐专业**的冲/稳/保叶子列表，按往年等效位次分档。访客位次存浏览器本地（localStorage、不上传），可在院校/专业页以冲稳保角标复用。

**已知取舍（为何逐专业而非按组）**：平行志愿实际以院校专业组为填报单位，逐专业输出更细但不直接等于填报单位、且组内调剂风险在纯逐专业视图里看不出来。缓解：每条专业叶子须标注其 2026 所属院校专业组 + 组内调剂提示，使逐专业视图仍能落到实际填报动作。
